### 1.1 静态分析

| 检查项 | 工具/方法 | 标准 |
|--------|----------|------|

### 1.2 代码审查清单

| 维度 | 检查点 | 状态 |
|------|--------|------|
| **异步安全** | async 后是否 `if (!mounted) return` | |
| **状态管理** | Provider/Notifier 是否正确 dispose | |
| **数据库** | 所有数据库操作是否有异常处理 | |
| **网络** | HTTP 请求是否有超时和错误兜底 | |
| **UI** | 是否所有颜色从 Theme 获取（非硬编码） | |
| **导航** | 是否使用统一的路由方式（非 Navigator.pushNamed） | |
| **资源** | 图片/大对象是否及时释放 | |
| **多语言** | 硬编码中文字符串是否应提取 | |

**审查要求**：每个文件至少一人审查。关键模块（数据库、网络、认证）需两人交叉审查。

### 1.3 依赖审计

```bash
# 检查过期依赖
flutter pub outdated

# 检查已知漏洞
dart pub audit --help  # Dart 3.6+ 如有此命令可用
```

| 检查项 | 标准 |
|--------|------|
| 所有依赖是否最新 stable 版本 | 小版本可接受，大版本评估迁移成本 |
| 是否有未使用的依赖 | pubspec.yaml 中每个包都在代码中被 import |
| 是否有已知 CVE 漏洞 | 至少手动检查关键依赖（dio, drift, sqlite3） |

### 1.4 代码混淆验证

```bash
# 确保 ProGuard/R8 已启用
# android/app/build.gradle.kts:
#   isMinifyEnabled = true
#   isShrinkResources = true
```

| 检查项 | 说明 |
|--------|------|
| `isMinifyEnabled = true` | R8 混淆 + 优化已启用 |
| `isShrinkResources = true` | 移除未使用资源 |
| `proguard-rules.pro` | 已配置关键 keep 规则（如 json_serializable、riverpod 等） |
| 反编译验证 | `jadx` 打开 APK，确认关键类名被混淆为 a/b/c |

## 阶段九：发布与监控

### 9.1 发布流程

```
代码冻结 → 最终测试 → 签名构建 → 内部测试 → Alpha/Beta → 生产发布 → 监控
```

### 9.2 灰度发布策略

| 阶段 | 用户量 | 持续时间 | 观察指标 |
|------|--------|---------|---------|
| 内部测试 | 1-5 人 | 1-2 天 | 崩溃率、核心流程 |
| 封闭 Alpha | 10-50 人 | 3-5 天 | 反馈收集、Bug 修复 |
| 开放 Beta | 100-500 人 | 1-2 周 | 崩溃率、留存率、评分 |
| 生产发布 | 100% | — | 全面监控 |

### 9.3 崩溃监控

| 方案 | 适用场景 |
|------|---------|
| Firebase Crashlytics | Google Play 上架，免费 |
| Sentry | 自建/开源，支持 Flutter |
| Bugly（腾讯） | 国内应用市场，免费 |
| 自建日志 | 最小化方案，仅本地 logger |

**推荐**：国内用 Bugly，海外用 Crashlytics。

### 9.4 关键指标

| 指标 | 定义 | 健康值 |
|------|------|--------|
| 崩溃率 | 崩溃会话 / 总会话 | < 0.5% |
| ANR 率 | ANR 次数 / 总使用次数 | < 0.1% |
| 启动时间 | 冷启动到首页显示 | < 1.5s |
| 日活用户 (DAU) | 每日打开用户 | 持续增长 |
| 次日留存 | 安装次日再打开 | > 40% |
| 7 日留存 | 安装 7 天后仍用 | > 20% |

### 9.5 反馈渠道

| 渠道 | 说明 |
|------|------|
| 应用内反馈 | 设置页「反馈与帮助」入口 |
| 邮件 | `support@example.com` |
| 应用商店评论 | 每日查看并回复 |
| 用户群 | QQ/微信群 |

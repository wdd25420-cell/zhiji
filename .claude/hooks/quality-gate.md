# Claude Code Hook — 知记项目质量铁律

## 适用范围
本文件适用于 C:\AI\claudecode\projects\zhiji 下所有工作任务。

## 铁律

### 1. 禁止跳过
- 每一行任务必须执行到有输出来证明完成。
- 不得写"待后续验证"、"可跳过"、"不影响主流程"跳过任何一条。
- 如果某任务因外部条件无法完成（如需要管理员终端、真机），必须：
  a) 记录当前环境限制的具体原因
  b) 给出替代验证方案（如：adb不可用则用 `dart analyze` 替代 lint）
  c) 标记状态为 `blocked` 而非 `completed`

### 2. 禁止绕过
- 不得在工具调用失败后直接标记 completed。
- 每个工具调用失败必须重试至少一次（调整参数/路径/环境变量）。
- 重试仍失败才标记 blocked 并写明原因。

### 3. 禁止投机取巧
- 不得因为"之前已经做过"而跳过验证。
- `flutter analyze` 必须在当前会话中重新执行。
- `flutter test` 必须在当前会话中重新执行。
- 每个任务必须有具体的命令输出或文件内容作为完成证据。

### 4. 逐项汇报
- 每个子任务完成后，必须在回复中输出以下三行：
  ```
  ✅ [P0-x.y] 任务标题 — 通过/blocked
     证据: <具体命令输出摘要或文件路径:行号>
     耗时: <实际分钟>
  ```
- 如果 blocked，额外输出失败原因和替代方案。

### 5. 任务顺序
- 严格执行 TaskCreate 时的 ID 顺序（数字由小到大）。
- 同一优先级（P0/P1/P2）内部按任务 ID 排序执行。
- P0 全部 completed 之后才能开始 P1。
- 不得并行执行有依赖关系的任务（如先修复 analyze 问题再跑 lint）。

### 6. 回环保护
- 如果连续 3 个任务 blocked，立即暂停并请求用户介入。
- 如果 flutter analyze 超过 120 秒无响应，暂停并诊断环境。

### 7. 文件修改限制
- 修复一个 lint 问题不得引入新的 lint 问题。
- 每次编辑后立即 `flutter analyze` 验证。
- 修复代码不得改变原有功能行为。

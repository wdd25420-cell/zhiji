## 一、我的角色与工作方式

### 1.1 我在这个项目中的定位

这个项目从头到尾由我主导。AI 是效率工具，不是替代者。

| 我负责 | AI 负责 |
|--------|--------|
| 产品方向与功能定义 | 根据我的指令编写代码 |
| 技术选型最终决策 | 提供选型对比和推荐方案 |
| 质量验收——《能不能交付》 | 自动化验证——《编译过不过》 |
| 叫停与纠正——《这个不行，重做》 | 根据我的反馈重新执行 |
| 环境操作——《管理员终端、签名、构建》 | 提供操作命令 |
| 审核标准制定——《不能跳过、不能绕过》 | 逐项执行审核流程 |
| 面试材料方向——《以我为主》 | 按要求整理输出 |

### 1.2 我给 AI 定的规矩

我在项目过程中反复纠正 AI 的惰性倾向，最终形成了一套硬性规则，写入了项目的 hook 文件：

**规则一：不能跳过**
> "一步步执行，不可以绕过。如果跑不通一些流程需要人工介入，就停下项目等待人工操作完成。"

**规则二：不能批量修复**
> "你不能批量修复，你必须逐一修复。" —— 批量修容易引入新问题，逐个修+逐个验证才是工程方法。

**规则三：必须有证据**
> "你都没完整全面的审核，断绝这种偷懒的操作，必须一步一步落实到位，你先给自己列一个任务清单，然后一项一项执行。"

**规则四：先审查再行动**
> "你需要进行重新审查，再进行修复。" —— 不搞清楚问题全貌就动手，属于低效开发。

**规则五：持续执行直到完成**
> "你一直执行到任务跑完。" —— 不允许半途停下。

**规则六：交付要真实**
> "AI 智能问答功能开发完了？你有验证吗？不能擅自跳过，不能给我一个空项目。" —— 功能不是「写了代码」就完成了，必须验证 API 链路、数据链路、UI 渲染全部通过。

### 1.3 我的质量信念

我始终认为**和 AI 协作不是「让 AI 替我干活」，而是「我指挥 AI 按照我的标准干活」**。如果我的标准模糊，AI 就会给模糊的输出。如果我放松要求，AI 就会走捷径。

整个项目中，我至少主动叫停并纠正 AI 7 次。每一次都直接提升了最终交付质量。

## 二、项目从零到一：我如何主导每一步

### 2.1 第一天：从"要做什么"到"能跑起来"

#### 我提出方向

我最开始的需求很明确：我要做一个移动端 APP。在和 AI 的讨论中，我逐步明确了技术路线：

- 平台：Android
- 框架：Flutter（跨平台潜力）
- 应用类型：AI Agent 驱动的个人知识管理
- 产品名：知记 (Zhiji)

AI 给出了几个方案，**我最终拍板 Flutter**——虽然我当时没有 Flutter 经验，但我判断 Flutter 的跨平台能力和 Material Design 3 的组件体系更适合这个项目。

#### 我亲自解决环境问题

Flutter 开发环境在 Windows 11 24H2 上遇到严重障碍：Gradle 构建持续崩溃。

AI 分析了错误日志，判断是 Win11 24H2 内核的 `Pipe.open()` 实现变更导致所有标准 JDK 的 Selector 崩溃。AI 给出了排查思路，但具体需要我操作：

- 我安装了 Android Studio
- 我在管理员 PowerShell 中执行了 `netsh int ipv4 set global loopbacklargemtu=disable`
- 我配置了 JBR JDK 21 的环境变量
- 我安装了 BlueStacks 作为替代模拟器方案

**这个阶段的关键认知**：AI 可以诊断问题，但不能操作我的电脑。环境问题必须我来解决。如果我只是等着 AI 给出一个"能用的命令"，项目第一天就会卡死。

#### 我确立了架构设计

AI 给出了完整的技术选型对比表。我审核了每一项决策：

| 决策 | AI 推荐 | 我的判断 |
|------|--------|---------|
| 状态管理 | Riverpod | 同意。无 BuildContext 依赖，测试友好 |
| 数据库 | drift (SQLite) | 同意。需要 FTS5 全文搜索，原生 SQLite 最可靠 |
| 路由 | GoRouter | 同意。声明式 DSL 比命令式清晰 |
| HTTP | Dio | 同意。需要拦截器和超时控制 |
| AI SDK | 直接调 DeepSeek API | 同意。不需要 LangChain 等重量级框架 |

我让 AI 先写计划再执行，每一个 Phase 必须有验收标准。这是我从软件工程实践中带过来的习惯——**没有验收标准的开发等于没有开发**。

#### 我发现并推动修复了第一个关键 Bug

功能开发完成后，我亲自点击测试。发现"写日记"按钮完全没反应。

我立即反馈给 AI，AI 排查后发现是 GoRouter 的 `StatefulShellRoute` 与 `Navigator.pushNamed` 不兼容。全项目有 13 处使用了错误的导航方式。

我要求 AI 逐一替换、逐一验证。不能搜索-替换一把梭。

#### 我要求建立持续开发机制

系统重启后，AI 的会话上下文会丢失。我直接质问：

> "重启之后你怎么能保证你接着项目继续干？？"

这促使 AI 创建了 `CLAUDE.md` 文件——完整记录项目架构、数据模型、路由表、环境配置。之后每次新会话，AI 都能从这个文件快速恢复上下文。**这是我主动推动的基础设施建设**。

#### 我提供的 DeepSeek API Key——安全考量

在代码审查阶段，我提供了 DeepSeek API Key。但我明确要求 AI：

- 不能把 Key 写在代码文件里
- 必须加密存储

AI 使用了 `FlutterSecureStorage`——底层对接 Android Keystore 硬件加密。这意味着即使我的设备被 root，Key 也不会泄露。Git 仓库中没有任何 Key 的痕迹。

### 2.2 第二天：从"能跑"到"能交付"

#### 我推动功能对标行业标准

我不满足于"功能都在"的状态。我让 AI 去搜索合格的 Android App 需要具备什么，然后对比我的项目进行差距分析。

AI 找到了 26 条改进点。我逐一审核，确定了优先级。这个过程中我补充了：

- 语音输入（中文）——我要求支持，因为移动端输入体验很重要
- 数据可视化（情绪趋势图、写作热力图）——空白的首页不够有说服力
- AI 三件套（续写/润色/总结）——光有分析不够，要有创作辅助

#### 我推动实现了四个复杂功能模块

**AI 智能问答（RAG）**

我看到项目计划中这个功能被标为"P3 — 跳过"。我的判断是：AI 问答是这个产品最亮眼的差异化功能，不能跳过。我问 AI："这个功能可以利用 API 进行对话问答吗？"

AI 确认可行后，我说了一个字："做。"

完成后 AI 汇报"已开发完"。我立刻追问验证结果——我不接受口头报告，我要看到证据。AI 随后逐项列出了 API 链路、数据链路、UI 层、路由入口的完整验证结果。

**批量删除**

我主动提出需要这个功能。移动端列表管理，不能一个个删。AI 实现后我自己测试了长按、多选、全选、取消、删除确认的完整交互流程。

**附件系统**

我要求支持拍照、相册、Word、PDF 等主流文件格式。这是知识管理应用的基本功——用户需要把外部资料导入到知识库中。

AI 设计了完整的 `FileAttachmentManager` 系统，支持 24 种文件格式。我在审核中发现权限引导不够完善，要求补充了相机和相册权限拒绝时的引导弹窗。

**撤销/重做**

我看到评估报告中这被列为"未实现"，直接要求开发。AI 设计了 UndoManager 快照栈系统。我特别关注了快照触发时机——不能每个按键都快照（浪费内存），也不能丢了关键操作。最终方案是 1.5 秒防抖 + AI/Markdown/语音/文件操作前手动快照。

#### 我发现了最隐蔽的 Bug

在测试附件功能时，我发现一个严重问题：添加附件后，之前编写的内容全部丢失。

这不是传统的逻辑 bug——是 Android 系统级行为。当 App 启动相机/相册/文件选择器，系统可能在内存不足时杀掉后台的 Flutter 进程。进程重建后，`State` 对象中的 `TextEditingController` 被重置，所有未保存输入消失。

AI 提出了草稿保护方案——启动外部 Activity 前将全部字段 JSON 序列化到数据库，重建后自动恢复。我认可了这个方案并推动在两个编辑器中都实现了。

**这个 Bug 的发现完全依赖用户测试。自动化测试无法覆盖系统杀进程这种场景。**

#### 我指挥了三轮代码审查

这是我整个项目中最下功夫的地方。AI 有惰性——会倾向于给出模糊的、缺少证据的评估。我三次纠正了这个问题：

**第一轮**：AI 做了基础审查，输出 33 个问题。成果不错。但我强调"不能批量修复，逐一修复"。

**第二轮**：AI 给了我一份"上线前评估报告"，评分 55%。我一眼看出问题——报告没有代码级证据，属于印象派评估。我直接说：

> "我看你都没完整全面的审核，断绝这种偷懒的操作，重新进行审查，必须一步一步落实到位，你先给自己列一个任务清单，然后一项一项执行；然后给自己的hook文件里规定不能跳过、不能绕过。"

AI 重新设计了 8 步系统化审计流程，每一步都有具体的 grep 命令和行号。最终发现了 16 个真实问题（P0 编译错误 2 + P1 安全漏洞 8 + P2 优化 6）。评分从 55% 下调到 39%——这才是真实的水平。

**第三轮**：构建前的最后一次审查。只剩 2 个问题——说明前两轮修复已经覆盖了绝大多数问题。质量在收敛。

每一次审查-修复循环都在提升代码质量。测试从 42 增长到 67，全部通过。`flutter analyze` 始终保持 0 error 0 warning。

#### 我完成了最后的构建与发布

Release APK 构建只能在管理员 PowerShell 中执行——这是 Windows 11 的环境限制。我亲自执行了构建命令。

第一次构建报签名文件找不到。AI 排查发现 `key.properties` 的 `storeFile` 路径写错了（相对路径基准目录是 `android/app/` 而非 `android/`）。我确认修复后重新构建，3 个 APK 成功产出。

然后我在 GitHub Desktop 上创建了仓库、推送了代码、创建了 v1.0.0 Release、上传了 3 个 APK 安装包。

整个发布流程是在我电脑上由我操作完成的。AI 提供命令，我执行。

## 三、我的关键决策与干预——7 个转折时刻

### 决策 1：坚持先写计划再执行（Day 1 上午）

AI 想直接开始写代码。我阻止了——要求先写完整的 6 阶段实施计划，每个 Phase 必须有明确的**验收标准**。

**影响**：没有这份计划，后续的开发会变成散乱的代码堆砌。计划提供了进度跟踪的坐标系。

### 决策 2：推动环境问题的深度排查（Day 1 上午）

Gradle 构建失败时，AI 最初建议换个 JDK 试试。我没有接受"试试"这种模糊方案——我要求 AI 解释根因。这促使 AI 深入分析 Win11 24H2 内核变更，最终找到了 JBR JDK 21 这个可行方案。

**影响**：如果只是"换个 JDK 试试"，可能试 5 个都失败，浪费大量时间。深入理解根因才能精准解决。

### 决策 3：要求建立上下文持久化机制（Day 1 下午）

重启后 AI 丢失上下文，这是 AI 辅助开发的致命问题。我直接质问并要求解决。结果是 `CLAUDE.md`——项目级别的 AI 记忆文件。

**影响**：没有这个文件，每次重启要花 30 分钟重新向 AI 解释项目背景。现在 AI 加载这个文件就能直接开始工作。

### 决策 4：坚持逐项修复而非批量修复（Day 1 傍晚）

AI 想批量修复 33 个审查问题。我拒绝了——要求逐一修复、逐一验证。理由是：批量修如果引入新问题，很难定位是哪个修改导致的。

**影响**：33 个问题的修复过程零回滚，`flutter test` 从 42 稳定增长到 67。

### 决策 5：拒绝模糊的评估报告（Day 2 下午）

AI 给了第一版评估报告后，我判断它不够详实——没有代码行号、没有 grep 证据、没有命令行输出。我直接要求重做。

**影响**：第二版报告才发现 2 个编译错误（voice_input_button 的误修改）和 1 个安全风险（allowBackup=true 导致数据库可被备份提取）。这些在第一版的"印象评估"中全部漏掉了。

### 决策 6：坚持 RAG 功能不跳过（Day 2 下午）

项目计划中 AI 问答被标为 P3（最低优先级）。我的判断不同——这是产品最亮眼的差异化功能。我推动开发，完成后又亲自追问验证结果。

**影响**：RAG 问答成为项目面试展示的核心亮点之一。

### 决策 7：发现并要求修复草稿丢失 Bug（Day 2 傍晚）

这是自动化测试无法覆盖的场景——系统杀进程。我是在实际使用中发现的问题。如果我只看 AI 的测试报告"67/67 passed"，这个 Bug 就会留在 Release 版本里。

**影响**：草稿保护机制成为项目技术亮点。面试中可以展开讲这个 Bug 的发现过程和解决方案。

## 五、我对 AI 辅助开发的认知总结

### AI 能做什么

- 快速编写大量标准化代码（CRUD、UI 布局、配置）
- 提供技术选型对比和推荐
- 自动执行静态分析和单元测试
- 扫描代码中的常见模式问题
- 生成结构化文档

### AI 不能做什么

- 理解真正的用户需求（它只能理解我明确表达的）
- 判断优先级（P3 的功能可能是杀手锏，需要我来判断）
- 保持质量一致性（需要我持续监督和纠正）
- 操作物理设备（构建、安装、真机测试需要我）
- 在上下文丢失后自动恢复（需要我提前建立机制）

### 我的核心经验

1. **标准决定质量**。我对 AI 的要求精度直接决定了输出质量。模糊的指令得到模糊的结果。
2. **验证不能外包**。AI 可以跑测试，但不能替代我来判断这个功能是否真的可用。
3. **主动干预比被动接受更重要**。我 7 次叫停 AI，每次都推动了质量提升。
4. **基础知识决定上限**。我能理解 AI 的技术方案，能做选型判断，才能在 AI 偏离时纠正。
5. **环境问题只能自己解决**。AI 给命令，我执行。构建环境、签名、发布——这些需要真人操作。
